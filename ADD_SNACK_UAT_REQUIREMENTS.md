# 添加零食页面 UAT 需求补充

## 背景

UAT 反馈指出，当前添加零食流程里有两处会增加重复输入成本：

- 品牌需要用户完全手动填写，不够高效。
- 每次添加零食都要手动输入宠物名称，不适合多宠家庭长期使用。

本次需求聚焦添加零食页面的信息录入效率和宠物信息复用。

## 需求 1：零食品牌选择与自动填充

### 目标

用户添加零食时，可以从常见宠物零食品牌中快速选择品牌，也可以输入自定义品牌。

### 功能说明

- 在添加零食页面新增 `Brand` 输入项。
- `Brand` 支持搜索、选择和自定义输入。
- 用户输入时展示品牌建议列表。
- 选择建议品牌后自动填充品牌字段。
- 如果用户输入的品牌不在建议列表中，允许保留自定义内容。
- 编辑已有零食时，需要回显已保存品牌。

### 初始品牌建议

建议首版内置常见品牌，后续可继续扩展：

- Temptations
- Greenies
- Churu
- Sheba
- Fancy Feast
- Wellness
- Blue Buffalo
- Purina
- Royal Canin
- Hill's
- Ziwi Peak
- Feline Natural
- KONG
- Milk-Bone
- Pedigree
- Other

### 验收标准

- 用户可以通过输入关键字筛选品牌。
- 用户点击品牌建议后，品牌字段自动填入。
- 用户可以输入并保存自定义品牌。
- 保存后的品牌在零食列表或详情中可查看。
- 语言切换不应翻译用户输入的品牌名称。

## 需求 2：产品名选择、自动填充与扫码录入

### 目标

用户添加零食时，产品名不应只能手动输入。系统应支持常见产品名建议、品牌联动筛选、自定义输入，并探索通过扫描包装二维码或条形码自动填充产品信息。

### MVP 方案选择

MVP 阶段只采用免费方案：

- 使用内置品牌和产品建议库实现自动补全。
- 使用原生能力或免费 SDK 做本地条码/二维码识别。
- 优先尝试免费开放产品库做命中率验证。
- 不接入付费商品查询 API。
- 如果免费产品库查询不到结果，回退到手动录入。

### 功能说明

- 添加零食页面的 `Snack name` 支持自动补全。
- 用户输入产品名关键字时展示建议列表。
- 当用户已选择 `Brand` 后，产品名建议优先展示该品牌下的产品。
- 用户选择产品建议后，自动填充：
  - Snack name
  - Brand
  - Category
  - 可选：产品图片、规格、默认单位
- 如果用户输入的产品名不在建议列表中，允许保留自定义内容。
- 编辑已有零食时，需要回显已保存产品名。
- 扫码入口建议放在 `Snack name` 或 `Brand` 附近，作为快速录入方式。

### 扫码录入范围

首版建议同时考虑：

- 一维商品条码：UPC-A, UPC-E, EAN-8, EAN-13
- 二维码：QR Code

扫码后系统根据识别结果进行处理：

- 如果扫描到商品条码：使用条码值查询产品数据库，并自动填充产品名、品牌和分类。
- 如果扫描到二维码 URL：尝试解析 URL 中的产品信息；如无法解析，则仅保存或展示链接，不强行填充。
- 如果查询不到产品：展示“未找到产品”，并保留手动填写入口。
- MVP 不承诺通过扫码自动识别保质期；过期日期仍由用户手动选择。

### 保质期自动识别决策

MVP 阶段不把“扫码自动填充保质期”作为核心能力。

原因：

- 普通商品条码 UPC/EAN 通常只标识产品，不包含具体保质期。
- 包装二维码内容不统一，可能是官网、营销页、防伪页或溯源码，无法稳定解析保质期。
- 保质期通常以包装上的文字形式出现，例如 `EXP 2026/09/30`、`Best Before 30 Sep 2026`，需要 OCR 识别而不是普通扫码。

MVP 处理方式：

- `Scan barcode` 仅用于尝试自动填充产品名、品牌和分类。
- `Expiry date` 仍使用日期选择器手动录入。
- 扫码命中产品后，可以把用户注意力引导到过期日期字段。

后续版本探索：

- V1 可增加 `Scan expiry date` 实验入口。
- 使用 OCR 识别包装上的 `EXP`、`Best Before`、`Use By` 等日期文字。
- 识别结果需要用户确认后才写入过期日期。
- 如果 OCR 识别不确定，应展示候选日期或回退手动选择。

### 验收标准

- 用户可以通过输入关键字筛选产品名。
- 产品名建议可以根据已选品牌缩小范围。
- 用户点击产品建议后，产品名字段自动填入。
- 用户可以输入并保存自定义产品名。
- 扫码成功后，如果数据库命中，应自动填充产品名和品牌。
- 扫码失败或数据库未命中时，用户仍可手动添加。
- 语言切换不应翻译用户输入的产品名。

### 推荐实现分层

#### MVP

- 内置一份常见品牌和常见产品名列表。
- 支持品牌选择后联动产品名建议。
- 支持自定义品牌和产品名。
- 可做本地扫码识别，但不依赖付费产品数据库。
- 可接入免费开放 API 做验证，例如 Open Food Facts；如果命中率不足，不阻塞主流程。

#### V1

- 增加扫码入口。
- 本地识别条码或二维码。
- 接入 1 个产品查询 API。
- 查不到时允许用户手动填写，并可将本地记录缓存为个人常用产品。
- 可探索 OCR 识别包装上的保质期文字，但不影响 MVP 验收。

#### V2

- 建立自有产品库。
- 支持用户贡献产品信息。
- 支持按地区维护不同品牌和产品条目。
- 支持产品图片、规格、适用宠物类型、开封后建议食用期限。

## 需求 3：宠物档案与宠物选择

### 目标

如果添加零食页面保留宠物字段，则需要支持宠物档案管理，避免用户每次重复输入宠物名称。

### 功能说明

- 新增宠物管理入口。
- 用户可以添加、编辑、删除家里的宠物。
- 添加零食时，`Pet` 字段从手动输入改为选择已有宠物。
- 如果用户还没有添加宠物，添加零食页面应提供快捷入口去添加宠物。
- 允许零食不绑定宠物，适合多宠共享零食。
- 支持一个零食绑定多只宠物，适合家中多只宠物共享同一款零食。

### 宠物字段建议

MVP 可先保留轻量字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| Pet name | 是 | 宠物名称 |
| Species | 否 | Cat, Dog, Other |
| Avatar color | 否 | 用颜色区分宠物 |

### 页面入口建议

- `Settings` 页面增加 `Pets` 管理入口。
- 添加零食页面的 `Pet` 字段旁增加 `Manage` 或 `Add pet` 入口。
- 如果没有宠物，`Pet` 区域展示空状态和 `Add pet` 按钮。

### 验收标准

- 用户可以创建至少 1 个宠物档案。
- 添加零食时可以从已有宠物中选择，且支持多选宠物。
- 用户不需要每次重复输入宠物名称。
- 删除宠物时，需要处理已绑定零食：
  - MVP 不禁止删除有关联零食的宠物。
  - 删除后从相关零食的宠物绑定中移除该宠物。
  - 如果某个零食没有剩余绑定宠物，则变为未分配 / 共享零食。
- 语言切换不应翻译用户输入的宠物名称。

## 对当前原型的影响

### 添加零食页面

- 新增 `Brand` 字段。
- `Snack name` 从普通文本输入升级为支持自动补全的输入框。
- 新增扫码入口，用于扫描商品条码或二维码。
- `Pet` 字段从文本输入升级为多选控件。
- 无宠物时提供添加宠物入口。

### 设置页面

- 新增 `Pets` 管理入口。
- 进入后可查看、添加、编辑、删除宠物。

### 数据模型

新增 `Pet` 模型：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 唯一 ID |
| name | string | 宠物名称 |
| species | enum | cat, dog, other |
| avatarColor | string | 宠物标识颜色 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

更新 `Snack` 模型：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| productName | string | 零食产品名，支持建议产品或自定义 |
| brand | string | 零食品牌，支持建议品牌或自定义 |
| barcode | string | 扫描得到的 UPC/EAN/GTIN 条码，可为空 |
| productUrl | string | 扫描二维码得到的产品链接，可为空 |
| petIds | string[] | 关联宠物 ID 列表，可为空数组 |

新增 `ProductSuggestion` 模型：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 唯一 ID |
| productName | string | 产品名 |
| brand | string | 品牌 |
| category | string | 分类 |
| barcode | string | UPC/EAN/GTIN 条码 |
| imageUrl | string | 产品图片，可为空 |
| source | enum | builtin, api, user |

## 扫码与产品库 Research

### 需要研究的问题

- 目标市场：第一版主要覆盖澳洲、美国、中国，还是全球？
- 包装码类型：宠物零食包装上 UPC/EAN 商品条码和二维码各自占比多少？
- 数据覆盖率：通用商品库对宠物零食、猫狗零食、进口品牌的命中率如何？
- 产品字段质量：API 是否稳定返回产品名、品牌、分类、图片和规格？
- 地区差异：同一品牌在不同国家是否使用不同条码和产品名？
- 隐私合规：相机权限文案、扫码数据是否上传、是否需要用户同意。
- 离线策略：无网时是否只识别条码，等联网后再查询产品信息。
- 兜底体验：扫码命中失败后，如何让用户继续快速手动添加。

### 候选技术方案

| 模块 | 候选方案 | 初步判断 |
| --- | --- | --- |
| iOS 扫码 | Apple VisionKit DataScannerViewController | 原生体验好，可识别文本、条码和二维码；需要相机权限 |
| 跨平台扫码 | Google ML Kit Barcode Scanning | 支持常见一维和二维码，本地识别，不需要网络 |
| 保质期 OCR | Apple Vision / VisionKit Text Recognition | 可识别包装上的日期文字；需要设计用户确认流程 |
| 保质期 OCR | Google ML Kit Text Recognition | 可做跨平台文字识别；日期格式解析和误识别需要额外处理 |
| 产品查询 | Open Food Facts API | 免费开放数据，但偏人类食品，宠物零食覆盖率需要验证 |
| 产品查询 | UPCitemdb API | 有免费额度和付费档位，适合验证 UPC/EAN 查询 |
| 产品查询 | Barcode Lookup API | 付费商业库，返回字段较全，但需要评估成本和宠物品类覆盖 |
| 自有产品库 | 内置 + 用户贡献 + 后台维护 | 长期最可控，但需要运营和数据维护 |

### 预算预估

#### 仅做本地自动补全

- 外部 API 成本：$0
- 适用阶段：MVP
- 风险：产品覆盖依赖我们手动维护的内置列表。

#### 扫码识别，不查外部产品库

- 扫码 SDK 成本：$0，使用原生能力或 ML Kit。
- 外部 API 成本：$0
- 适用阶段：技术验证。
- 风险：只能得到条码或二维码内容，不能自动知道产品名和品牌。

#### 扫码 + 免费产品查询

- Open Food Facts：免费开放 API。
- UPCitemdb 免费档：100 次请求/天。
- 适用阶段：小范围 UAT 或 demo。
- 风险：免费额度有限，宠物零食命中率未知。

#### 扫码 + 付费产品查询

- UPCitemdb DEV：约 $99/月，20,000 lookup/天。
- Barcode Lookup Starter：约 $99/月，5,000 API calls/月。
- Barcode Lookup Advanced：约 $249/月，25,000 API calls/月。
- 适用阶段：MVP 不采用；如果 UAT 证明扫码是核心体验，再进入付费验证。
- 风险：即使付费，也不保证宠物零食覆盖率足够，需要先抽样测试。

### MVP 推荐实现

1. 添加页先实现 `Brand` 和 `Snack name` 的本地自动补全。
2. 建立一份内置 `ProductSuggestion` JSON，字段包含品牌、产品名、分类、可选条码。
3. 扫码能力作为快速录入入口，但首版只保证能识别条码/二维码。
4. 扫到条码后先查本地内置产品库。
5. 本地未命中时，可尝试免费开放 API。
6. 仍未命中时，提示用户手动填写，并保留扫描到的条码。
7. 用户手动保存后，将该产品加入“我的常用产品”，后续可自动补全。

### 推荐 Research 计划

1. 收集 30-50 个真实宠物零食包装条码，覆盖猫、狗、冻干、罐头、肉干、磨牙、营养膏。
2. 用 Open Food Facts、UPCitemdb、Barcode Lookup 分别查询命中率。
3. 记录每个 API 返回字段完整度：产品名、品牌、分类、图片、规格。
4. 计算 UAT 规模下的调用量：日活用户、平均添加零食次数、扫码占比。
5. 根据命中率和成本决定 V1 是否接入外部 API，或先做内置产品库。
6. 另行收集 20-30 张真实包装保质期照片，验证 OCR 对 `EXP`、`Best Before`、`Use By` 等格式的识别准确率。

## 开放问题

- 添加零食时是否允许选择多个宠物？
- 是否需要支持“全家宠物共享”的零食标签？
- 品牌建议列表是否按猫/狗区分？
- 删除宠物时，已关联零食应该禁止删除、自动解绑，还是提示用户重新分配？
- 产品名建议是否需要按宠物类型区分，例如猫零食和狗零食？
- 扫码入口是否放在添加页顶部，还是只放在产品名字段右侧？
- 如果二维码跳转到品牌官网，是否允许打开外部链接，还是只提取信息？
- 是否需要让用户把扫码未命中的产品保存为“我的常用产品”？
- V1 是否需要单独提供 `Scan expiry date` 入口，还是等产品扫码流程稳定后再做？
