# 宠物零食保质期提醒小工具 PRD

## 1. 产品概述

### 1.1 产品名称

暂定名：Pet Snack Tracker

### 1.2 产品定位

面向养宠家庭的原生移动端轻量工具，用于记录家中宠物零食库存、购买日期和过期时间，并在零食临近过期时主动提醒，帮助用户减少浪费、避免误喂过期食品。

### 1.3 平台与语言

- 第一版采用原生移动端开发，优先面向手机使用场景。
- 应用默认语言为英文。
- MVP 需要支持应用内语言切换。
- 首批语言建议支持 English 和 简体中文。
- PRD 使用中文维护，应用内展示文案以英文为主，并提供中文翻译资源。

### 1.4 核心场景

用户购买宠物零食后，打开小工具快速录入零食信息和过期时间。之后用户可以随时查看家中还有哪些零食、哪些即将过期、哪些已经过期。当零食进入临期范围时，系统向用户发送提醒，促使用户优先消耗或清理。

## 2. 背景与问题

养宠家庭常会囤积多种宠物零食，例如冻干、罐头、磨牙棒、肉干、营养膏等。零食往往散落在厨房、储物柜、宠物用品箱中，用户容易忘记具体购买时间和保质期。

当前痛点：

- 宠物零食种类多，靠记忆管理容易遗漏。
- 包装袋拆封后过期风险更高，但用户不容易持续追踪。
- 临期零食没有明显提示，容易造成浪费。
- 已过期零食可能被误喂，影响宠物健康。
- 现有备忘录或日历提醒录入成本高，不适合频繁记录。

## 3. 目标用户

### 3.1 主要用户

有 1 只或多只宠物，并会定期购买宠物零食的养宠人。

典型特征：

- 经常网购或线下购买宠物零食。
- 会囤货，家里同时存在多包零食。
- 关心宠物饮食安全。
- 希望工具足够轻，不想维护复杂账本。

### 3.2 次要用户

多人共同照顾宠物的家庭成员，例如情侣、室友、父母与子女。多人共享管理可作为后续版本能力。

## 4. 产品目标

### 4.1 用户目标

- 快速记录新买的宠物零食。
- 清楚知道每个零食的过期时间和剩余状态。
- 在临期前收到提醒。
- 优先消耗临期零食，减少浪费。
- 避免误喂过期食品。

### 4.2 业务目标

- 建立一个低门槛、高频可用的移动端工具。
- MVP 聚焦记录和提醒闭环，不引入复杂社区、电商或健康管理功能。
- 为后续扩展宠物档案、购物清单、家庭共享和智能识别打基础。

## 5. 成功指标

MVP 阶段可关注以下指标：

- 新用户首次添加零食完成率。
- 单个零食平均录入耗时。
- 用户添加第二个零食的比例。
- 临期提醒打开率。
- 用户将临期零食标记为已吃完或已处理的比例。
- 7 日留存率。

## 6. MVP 范围

### 6.1 必须包含

- 编辑零食

## 7. 核心用户故事

### 7.1 添加零食

作为养宠人，我希望在买到新零食后快速记录名称、宠物、过期时间和数量，这样之后能知道家里还有哪些零食。

验收标准：

- 用户可以从首页进入添加页面。
- 用户必须填写零食名称和过期日期。
- 用户可以选择宠物、分类、数量、开封状态、提醒时间。
- 保存成功后，零食出现在列表中。
- 默认界面文案为英文。

### 7.2 查看临期零食

作为养宠人，我希望一打开应用就看到哪些零食快过期，这样可以优先喂掉它们。

验收标准：

- 首页按过期状态突出展示临期和已过期零食。
- 每条零食显示剩余天数。
- 已过期、临期、正常状态有明显区分。

### 7.3 收到提醒

作为养宠人，我希望系统在零食快过期前提醒我，这样我不会忘记处理。

验收标准：

- 用户可以设置默认提前提醒天数。
- 每个零食可使用默认提醒，也可单独调整。
- 到达提醒时间时，系统发送本地通知。
- 已删除或已吃完的零食不再触发提醒。

### 7.4 标记已吃完

作为养宠人，我希望能把已经吃完的零食标记掉，这样列表不会越来越乱。

验收标准：

- 用户可以将零食标记为已吃完。
- 已吃完零食默认不显示在当前库存列表。
- 用户可以在历史记录中查看已吃完零食。

## 8. 功能需求

### 8.1 首页 / 库存列表

功能说明：

- 展示当前仍在库存中的宠物零食。
- 默认按过期紧急程度排序：已过期、临期、正常。
- 支持按宠物、分类、状态筛选。
- 支持搜索零食名称。

列表字段：

- 零食名称
- 所属宠物
- 分类
- 数量
- 过期日期
- 剩余天数
- 开封状态
- 状态标签

状态规则：

- 已过期：当前日期晚于过期日期。
- 临期：距离过期日期小于等于用户设置的提醒天数。
- 正常：未进入临期范围。

### 8.2 添加 / 编辑零食

字段设计：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 零食名称 | 是 | 用户手动输入 |
| 所属宠物 | 否 | 可选择已有宠物，MVP 可先支持文本输入 |
| 分类 | 否 | 如冻干、罐头、肉干、磨牙棒、营养膏、其他 |
| 品牌 | 否 | 手动输入 |
| 数量 | 否 | 默认 1，可输入包、罐、条等单位 |
| 购买日期 | 否 | 默认当天 |
| 过期日期 | 是 | 用日期选择器录入 |
| 是否已开封 | 否 | 默认未开封 |
| 开封日期 | 否 | 开封后可填写 |
| 提醒设置 | 否 | 默认使用全局设置 |
| 备注 | 否 | 例如适口性、宠物喜好、存放位置 |

### 8.3 零食详情

展示内容：

- 零食基础信息。
- 过期状态。
- 距离过期还有多少天，或已过期多少天。
- 提醒设置。
- 备注。

可执行操作：

- 编辑
- 标记已吃完
- 删除
- 标记已开封

### 8.4 提醒设置

全局设置：

- 默认提前 7 天提醒。
- 用户可选择提前 1 天、3 天、7 天、14 天、30 天，或自定义天数。
- 用户可设置提醒时间，例如每天上午 9:00。

单个零食设置：

- 使用默认提醒。
- 自定义提前天数。
- 关闭该零食提醒。

通知文案示例：

- “Churu is expiring in 3 days. Feed it to Mochi first.”
- “Freeze-dried chicken expires today. Please check it soon.”
- “Dental chews expired 2 days ago. Review before feeding.”

### 8.5 历史记录

功能说明：

- 展示已吃完或已删除前归档的零食记录。
- MVP 中可仅支持“已吃完”历史。
- 历史记录默认按完成时间倒序排列。

### 8.6 语言切换

功能说明：

- 用户可以在设置页切换应用语言。
- 默认语言跟随系统语言；若系统语言不在支持范围内，则默认使用英文。
- 用户手动选择语言后，后续启动应用应保持该选择。
- 切换语言后，主要界面、按钮、状态标签、错误提示、空状态、通知文案均应使用目标语言。
- 用户输入内容不自动翻译，例如零食名称、宠物名称、备注保持原样。

MVP 支持语言：

- English
- 简体中文

关键英文文案示例：

| 场景 | English |
| --- | --- |
| 库存页标题 | Snacks |
| 添加入口 | Add Snack |
| 过期日期 | Expiry Date |
| 临期状态 | Expiring Soon |
| 已过期状态 | Expired |
| 正常状态 | Fresh |
| 已吃完 | Mark as Finished |
| 历史记录 | History |
| 设置 | Settings |
| 语言 | Language |
| 空状态 | No snacks yet |

## 10. 关键流程

### 10.1 首次使用流程

1. 用户打开应用。
2. 看到空状态和添加入口。
3. 点击添加零食。
4. 填写零食名称和过期日期。
5. 保存。
6. 返回库存列表，看到刚添加的零食。

### 10.2 临期提醒流程

1. 用户添加零食并设置过期日期。
2. 系统根据过期日期和提醒设置计算提醒时间。
3. 到达提醒时间后发送本地通知。
4. 用户点击通知进入零食详情。
5. 用户选择标记已吃完、编辑日期、或继续保留。

### 10.3 标记已吃完流程

1. 用户在列表或详情中点击已吃完。
2. 系统弹出确认。
3. 用户确认。
4. 零食从当前库存移至历史记录。
5. 该零食相关提醒取消。

## 12. 非功能需求

### 12.1 易用性

- 添加零食的核心路径控制在 30 秒内完成。
- 必填字段尽量少，默认只要求名称和过期日期。
- 日期选择器应适合单手移动端操作。
- 英文文案应简短直接，适配移动端小屏幕。

### 12.2 性能

- 首页列表在 100 条以内应即时加载。
- 本地搜索和筛选应无明显延迟。

### 12.3 可靠性

- 本地通知需要在保存、编辑、删除、标记已吃完时同步更新。
- 日期计算需考虑用户所在时区。
- 应避免重复通知同一条零食。
- 语言切换后，通知调度使用最新语言生成通知文案。

### 12.4 隐私

- MVP 可优先使用本地存储。
- 不需要收集宠物健康、用户地址或支付信息。
- 如后续加入云同步，需要补充账号体系和隐私说明。

### 12.5 本地化

- 所有面向用户的固定文案必须通过本地化资源管理，不应硬编码在页面中。
- 日期格式跟随当前应用语言和设备区域设置。
- 首批本地化覆盖 English 和 简体中文。
- 若某条文案缺少翻译，必须回退到英文。

## 13. 边界情况

- 用户输入过去日期作为过期日期：允许保存，但标记为已过期，并提示用户确认。
- 用户未授权通知：应用仍可使用，但提醒能力不可用；需展示开启通知的引导。
- 用户修改过期日期：重新计算并更新提醒。
- 用户关闭单个零食提醒：该零食不发送通知。
- 用户删除零食：取消该零食关联通知。
- 用户跨时区旅行：按设备当前时区计算提醒时间。
- 同一天有多个零食临期：可合并通知，也可逐条通知；MVP 建议逐条通知，实现更简单。
- 用户切换语言：界面立即更新；已调度通知需在下次保存或重新调度时使用新语言，MVP 可在切换语言后统一重建提醒。
- 系统语言不支持：默认使用英文。

## 14. 体验原则

- 快速：录入越快越好，不让用户像记账一样辛苦。
- 清楚：过期状态一眼可见，避免用户反复计算日期。
- 温和：提醒文案不制造焦虑，但明确提示风险。
- 轻量：MVP 不做复杂资产管理，专注“记下来”和“提醒我”。
- 可信：日期、状态、通知必须准确。

## 17. 开放问题

- MVP 是否需要支持多个宠物，还是先只记录宠物名称？
- 临期阈值默认设置为 7 天是否合适？
- 已过期零食是否默认留在库存列表顶部，还是移入单独的“需处理”区域？
- 是否需要记录开封后保质期？例如开封后 30 天内食用。
- 原生开发第一版优先 iOS、Android，还是双平台同时做？
- 语言切换是否只支持 English / 简体中文，还是还需要繁体中文？
